# ADR-002: データ取得戦略

## ステータス
Accepted

## コンテキスト
対象の6サービスはAPIの公開状況がバラバラ。統一的なデータ取得レイヤーを設計する必要がある。

## 決定

### レイヤー分け

| サービス | 取得方法 | 理由 |
|---------|---------|------|
| GMOコイン | Private REST API | 公式にAPIキー発行可能。残高照会エンドポイントあり |
| Coincheck | Private REST API | 同上。HMAC-SHA256認証 |
| オンチェーン | Alchemy/Etherscan API | アドレスさえあれば誰でも取得可能 |
| 楽天銀行 | Playwright スクレイピング | APIはfreee等のパートナー企業経由のみ。個人開放なし |
| 楽天証券 | Playwright スクレイピング | 公式APIは申請制かつ個人向けではない |
| PayPay | Playwright スクレイピング | APIは加盟店向け。残高照会は個人に開放されていない |

### スクレイピング設計方針
- 各スクレイパーは共通インターフェース `fetch_balances(config) -> list[Balance]` を実装
- Playwrightはヘッドレスモードで実行。デバッグ時のみheadful
- ログイン情報はconfig.yaml（gitignore対象）で管理
- 2FA対応: TOTP自動入力 or 手動介入待ちの設計を各サービスで検討
- レート制限: APIは公式レート制限に従う。スクレイピングは1日2-3回の低頻度で十分

### 認証情報の管理
- config.yaml をgitignoreに入れる
- config.example.yaml にキー名と説明だけ記載
- 将来的にはdotenvや環境変数に移行も可能

### エラーハンドリング
- 各スクレイパーは独立して失敗可能。1つ失敗しても他は実行される
- 失敗時はログ出力 + 前回取得値を保持（DBから最新を参照）
- スクレイピング対象のUI変更で壊れることを前提に、エラー検知を明確にする

## 結果
- API組は安定して取得できる
- スクレイピング組はUI変更リスクがあるが、低頻度実行なのでBANリスクは低い
- 共通インターフェースにより、新サービス追加時も同じパターンで実装できる
